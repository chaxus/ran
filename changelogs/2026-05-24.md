## 计划 (Planned)

### SSR 完整支持路线图

当前 `utils/ssr.ts` 的手写 mock 方案能输出 DSD HTML，但无法处理依赖真实布局计算的组件（`offsetWidth`、`getBoundingClientRect`）。完成以下所有阶段后，可覆盖主流 SSR 场景；canvas 组件（`r-colorpicker`、`r-radar`）仅输出占位，不在完整 SSR 范围内。

#### 现有基础（比预期更完整）

`utils/builder/` 模块已经提供了完整的 SSR 基础设施：

- **`isSSR`**（`env.ts`）— `typeof document === 'undefined'` 环境检测
- **`HTMLElementMock` / `ShadowRootMock`**（`mocks.ts`）— 完整 mock DOM，含 `serialize()` 输出 DSD HTML
- **`ElementBuilder`**（`core.ts`）— fluent builder，内置 `isSSR` 分支：SSR 下用 mock，浏览器下用真实 DOM；`serialize()` 已能产出带 `<template shadowrootmode="...">` 的完整字符串
- **`DeclarativeShadow`**（`factory.ts`）— 直接生成 DSD template 元素

现有组件（`r-button`、`r-progress`、`r-input` 等）已经通过 `Div()`、`View()` 等 factory 使用这套 builder 构建 shadow DOM，**结构层面的 SSR 能力已经就绪**。

这与 Lit 的核心思路一致：将"结构描述"与"DOM 副作用"分离，前者通过 `ElementBuilder` 链式调用表达，在任何环境都能运行和序列化。

#### 阶段一：CSS-first 重构（消除 JS 布局依赖）

影响最大、改动最小的优化，目标是让最常用组件变成真正 SSR 友好。

**`r-progress`**：当前 `updateUI` 读取 `offsetWidth` 来计算 dot 位置和 bar 缩放：
```typescript
// 现在
const width = this._progress.offsetWidth;
this._progressWrapValue.style.transform = `scaleX(${ratio})`;
this._progressDot.style.transform = `translateX(${width * ratio}px)`;
```
改为 CSS 变量驱动，宽度交给 CSS 引擎：
```typescript
// 目标
this.style.setProperty('--progress-percent', String(percent));
// CSS: .ran-progress-wrap-value { width: calc(var(--progress-percent) * 1%); }
// CSS: .ran-progress-dot { left: calc(var(--progress-percent) * 1%); transform: translateX(-50%); }
```

**`r-tabs`**：当前 `setTabLine` 用 `getBoundingClientRect` 计算指示线偏移：
```typescript
// 目标：改用 CSS 变量 --tab-index 和 --tab-width，由 CSS 计算 translateX
this.style.setProperty('--tab-active-index', String(index));
```

完成后这两个组件在 SSR 输出时只需要设置 CSS 变量，浏览器不需要任何 JS 就能正确渲染初始状态（no layout shift）。

#### ~~阶段二：引入 Happy DOM 替换手写 mock~~（已不必要）

> `utils/builder/mocks.ts` 里的 `HTMLElementMock` 已深度集成在 `ElementBuilder` 里，且 `addEventListener`、`classList`、`style`、`shadowRoot`、`querySelector` 等组件实际用到的 API 均已实现。引入 Happy DOM 会破坏现有集成，收益不明显。**跳过此阶段。**

#### 阶段三：按组件类型分层 SSR 策略

| 组件 | 策略 | 说明 |
|---|---|---|
| `r-button` / `r-input` / `r-checkbox` | 完全 SSR | 纯结构，CSS 驱动，阶段一后直接支持 |
| `r-progress` / `r-tabs` | 完全 SSR | 阶段一重构后支持，CSS 变量承载状态 |
| `r-select` / `r-modal` / `r-popover` | SSR shell | 输出关闭状态结构，交互在客户端激活 |
| `r-colorpicker` / `r-radar` / `r-scratch` | Placeholder SSR | SSR 输出占位元素（颜色 swatch / skeleton），canvas 在客户端渲染 |
| `r-player` | Poster SSR | SSR 输出 poster 图 + 静态控制栏 HTML，视频完全客户端接管 |

#### 阶段四（长期）：`renderToStream` API

现有 `ElementBuilder.serialize()` 和 `ShadowBuilder.serialize()` 已经能输出完整的 DSD HTML 字符串。在此基础上封装一个流式入口即可，**不需要重新实现序列化逻辑**：

```typescript
import { renderToStream } from 'ranui/ssr';

// Node.js HTTP handler
app.get('/', (req, res) => {
  const stream = renderToStream(`
    <r-button type="primary">Submit</r-button>
    <r-progress percent="65"></r-progress>
  `);
  stream.pipe(res);
});
```

内部实现为 `AsyncGenerator<string>`，解析 HTML 片段 → 实例化组件 → 调用已有 `serialize()` → yield 字符串块。静态部分立即 yield，含异步操作的组件等待后 yield。

#### 阶段五：`customElements` 注册机制

当前 `utils/ssr.ts` 的 `renderToString` 用手写 `tagMap` 映射类名到标签名，存在两个问题：新增组件需要手动维护，拼写错误静默失败。

目标是建立一套自动注册机制，在 SSR 环境下替代 `customElements.define()`：

```typescript
// utils/ssr-registry.ts
const registry = new Map<string, new () => HTMLElement>();

export function defineSSR(tagName: string, constructor: new () => HTMLElement) {
  if (isSSR) {
    registry.set(tagName, constructor);
  } else {
    customElements.define(tagName, constructor);
  }
}

export function getSSRConstructor(tagName: string) {
  return registry.get(tagName);
}
```

每个组件的注册语句从：
```typescript
customElements.define('r-button', Button);
```
改为：
```typescript
defineSSR('r-button', Button);
```

`renderToStream`（阶段四）内部解析到 `r-button` 标签时，通过 `getSSRConstructor('r-button')` 取到类，实例化后调用 `serialize()` 输出 DSD，无需手写 `tagMap`。

#### 阶段六：父子组件关系

`r-select` + `r-option`、`r-tabs` + `r-tab` 这类组合，SSR 下的执行顺序与浏览器不同：浏览器里父组件的 `connectedCallback` 能遍历已解析的子元素，Node.js 里组件是逐个实例化的，父组件拿不到子组件信息。

解决思路：在 `renderToStream` 的解析阶段，先构建完整的组件树，再自顶向下序列化，让父组件在 `serialize` 时能访问子节点：

```typescript
// renderToStream 内部：先建树，再序列化
const tree = parseToComponentTree(html);   // 建立父子关系
const output = serializeTree(tree);        // 父组件序列化时可读取 children
```

涉及组件：`r-select`（读取 `r-option` 的 value/label）、`r-tabs`（读取 `r-tab` 的 label/key）、`r-form`（读取表单字段）。

#### 阶段七：框架集成验证（Next.js）

完成以上阶段后，需要在 Next.js App Router 下做端到端验证：

- **Server Component** 中直接使用 `renderToStream` 输出 DSD HTML
- **Client Component** 中正常导入组件，DSD rehydration 在浏览器自动激活
- 验证 `'use client'` 边界对 Web Components 注册的影响
- 提供 `ranui/next` 适配入口，封装 Next.js 特有的配置（`transpilePackages`、`serverExternalPackages`）

> **浏览器兼容性**：仅针对 Chrome、Firefox、Safari 最新版，均原生支持 DSD（`<template shadowrootmode>`），不考虑 polyfill。

#### ~~阶段八（可选，按需评估）~~：构建时静态生成（SSG）

> **注**：此阶段仅影响 `r-colorpicker`、`r-radar`、`r-scratch` 三个 canvas 组件。这三个组件本身是交互工具或运行时数据可视化，SSR 价值有限；不做阶段五时退回阶段三的 Placeholder 策略即可，仅有轻微 CLS，可接受。代价（CI 装 Chromium、构建时间增加）高于收益，**建议默认跳过，有明确需求时再实施**。

对于真正无法避免布局计算的场景（如需要真实像素值的 canvas 组件），使用 Playwright 在构建阶段生成静态快照：

```typescript
// scripts/ssr-snapshot.ts（构建脚本，非运行时）
const browser = await chromium.launch();
const page = await browser.newPage();
await page.setContent(`<r-colorpicker value="#1677ff"></r-colorpicker>`);
await page.waitForFunction(() => customElements.get('r-colorpicker'));
const dsd = await page.evaluate(() =>
  document.querySelector('r-colorpicker')!.getInnerHTML({ includeShadowRoots: true })
);
```

适用于文档站、博客等静态场景，不适合运行时 per-request SSR。

## 使用说明

### 本地运行视觉回归测试

```bash
# 启动 dev server（如未运行）
pnpm --filter ranui dev

# 在另一个终端运行视觉测试（仅 Chromium）
pnpm --filter ranui exec playwright test --project=chromium test/e2e/visual/

# 查看截图报告
pnpm --filter ranui test:report
```

### CI 接入 Argos

1. 在 [argos-ci.com](https://argos-ci.com) 注册并关联 GitHub 仓库（开源项目免费）
2. 将获得的 token 添加至 GitHub Secrets：`Settings → Secrets → ARGOS_TOKEN`
3. 每次 PR 触发 `visual-regression.yml` 后，Argos 自动对比截图并在 PR 中评论差异报告
