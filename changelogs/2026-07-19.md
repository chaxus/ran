# Changelog — 2026-07-19

## packages/ranui

来自 weread UI 迁移（去 React，改 builder + signal）的实战反馈，做两处让 builder 更贴近
AI 直觉、并修一个 export 陷阱。

### builder：`children()` 支持响应式 getter（reactive children）

- **动机**：此前 `children()` 只接静态节点，反应式的「条件子节点 / 列表」必须手写
  `createRef` + `createEffect` + `replaceChildren`，且首帧 `ref.current` 为 null 需早退——
  样板重复、易错，也不是 AI 写响应式 UI 的第一直觉（AI 会直接写
  `.children(() => cond() ? A : B)`）。
- **改动**：`ElementBuilder.children()` / `replaceChildren()` 与 `ShadowBuilder.children()`
  的每个参数现在都可以是 **getter `() => 节点 | 节点[] | null`**。getter 标记一个 live 区域：
  读到的 signal 变化时 getter 重跑，区域 DOM 就地重建（锚点 comment 定位，静态兄弟节点位置不变）。
  绑定的 effect 由当前 `createRoot` 拥有，随页面 dispose 自动清理。
- **SSR**：getter 只求值一次（静态快照），反应式是纯客户端行为。
- 静态用法完全不变（新签名是旧签名的超集）；新增 4 个浏览器用例 + 2 个 SSR 用例。
- 类型：新增并导出统一的 `Child` 类型（`ranui/builder` 与 `@/utils/builder` 均导出），
  取代内联联合。**注意命名**：是 `Child`（children 的参数类型），与 `r-input` / `InputBuilder` 无关。

### builder：新增 `For` —— keyed 列表原语

- **动机**：上面的响应式 getter 子节点是「整块重建」，列表一变就丢掉行内 DOM 状态
  （焦点 / 滚动 / 非受控输入值 / 进行中的过渡），大列表还 O(n) 重建。keyed 列表要的是
  「按 key 复用节点、只动增删/移动的项」。
- **API**：`For({ each, key, render })`，Solid 风格。`each` 是响应式数组 getter；`key`
  必须唯一（用来把旧节点匹配到新项，这正是保留状态的前提）；`render(item, index)` 每项
  只跑一次，`index` 是 **getter**（reorder 后仍正确）。返回的句柄直接丢进 `children()`。
- **实现**：锚点 comment 定位；`Map<key, ItemScope>` 复用；每项一个 `createRoot` 独立作用域
  （行内 effect 随该项移除而 dispose，整表随所属 `createRoot` dispose，`onCleanup` 兜底）；
  DOM 重排只 `insertBefore` 那些位置不对的节点，复用节点保持 identity → 状态不丢。
- **SSR**：一次性快照渲染，不做 reconcile。
- 导出 `For` / `ForOptions` / `ForHandle`（两个 barrel 都有）；新增 6 个浏览器用例
  （初始渲染 / 节点复用 / reorder 保状态 / 删除 dispose / 响应式 index / 作用域 dispose）
  - 1 个 SSR 快照用例。
- **安全**：文本仍走 `createTextNode`（非 innerHTML），无 XSS 面；`key` 只作 Map 键，不入 DOM。

### builder：新增 `Show` —— 细粒度条件渲染

- **动机**：通用 getter 子节点是「读到的任一 signal 变了就整块重建」，对**条件**而言太粗——
  这恰恰是「一次 setState 重渲染整块」的老问题。SwiftUI/Solid 的模型是：body 只建一次，
  只有真正变化的「洞」更新。
- **API**：`Show({ when, children, fallback? })`。`when` 真值→`children`、假值→`fallback`；
  `children` 收到「已收窄真值」的 accessor，在绑定里读它即可就地更新、不重建。
- **实现**：把 `when` 的**布尔真值**包进 `computed`（memo 只在值变时通知），返回一个只读
  `on()` 的 getter → 复用既有响应式子节点机制，但**只在真假翻转时重建分支**，`when` 底层
  signal 变但真值不变时分支零重建。仅 ~5 行，无新挂载代码。
- 用例证明细粒度：`when: () => count() > 0`，`count` 从 1→2 时分支 build 计数保持 1、节点
  identity 不变、仅内部 `.text` 绑定更新；翻到 0 才 teardown。导出 `Show` / `ShowOptions`。
- **定位**：条件用 `Show`（细粒度）、集合用 `For`（keyed）、通用 getter 只留给「每次都换形状」
  的兜底。

### builder：补齐细粒度控制流套件 —— `Switch`/`Match` + `Index`

- **动机**：凑齐 Solid 同级的控制流原语，让「条件 / 列表」都有细粒度写法，通用 getter 只作兜底。
- **`Switch` / `Match`**（n 路条件，`Show` 的多分支版）：渲染首个 `when` 为真的 `Match`，
  否则 `fallback`。只 memo「命中分支的下标」→ 仅在命中分支切换时重建；求值在首个命中处短路
  （后续分支未命中时不订阅）。返回 getter，复用响应式子节点机制。
- **`Index`**（按**位置** keyed 的列表，`For` 的对偶）：下标 `i` 的节点跨更新复用、其 `item`
  是 signal（值变原地更新、节点不重建），节点从不移动。适合原始值数组 / 定长行；有稳定 id 且
  会 reorder 时用 `For`。分支句柄；SSR 一次性快照。
- 导出 `Index`/`IndexOptions`/`IndexHandle`/`Switch`/`SwitchOptions`/`Match`/`MatchClause`
  （两个 barrel）；新增 Index 3 例 + Switch 2 例（含细粒度断言）+ 4 个 SSR 快照。
- 至此控制流：值→getter 绑定；条件→`Show`/`Switch`；列表→`For`/`Index`；getter 仅兜底。

### theme-switch：修复 SSR 崩溃（构造器碰 document）

- **问题**：`<r-theme-switch>` 在构造器里用**裸 `document.createElement`** 建影子内容，
  `renderHTMLToString` 在 Node 实例化它时 `document is not defined` 直接崩——违反「自定义元素
  构造器不得碰 document」的规范。weread SSG 预渲染首页（含主题切换）时踩到。
- **修复**：改用 **isSSR-aware 的 builder**（`Div()`/`ButtonBuilder()`，同 `r-section` 的写法）
  建结构；SVG 图标 `innerHTML` 仅 client 设置（SSR 影子留空、client 重建，同 `r-icon`）。
- 新增 SSR 回归用例：`renderHTMLToString('<r-theme-switch>')` 不抛错、出 DSD + `ran-theme-switch`。

### 记录设计原则（防走偏）

- 在 `CLAUDE.md` builder 段顶部加了 **DESIGN INVARIANT** 硬性条款、`docs/BUILDER.md` 开头加了
  「build once, update in place」原则块：**view 只建一次,状态经细粒度绑定/原语更新,永不重跑
  view 整体重渲染**（即本库存在的意义——避开 React/JSX 模型）；新增 API 不得引入「状态变→整棵
  子树重跑」。

### builder：`For` 重复 key 加固（防泄漏）

- **问题**：`Map` 一个 key 只能存一个节点，重复 key 会让先出现的节点**失去追踪**（永不移除/
  dispose）→ 泄漏 + DOM 重复。
- **修复**：重复 key 直接忽略（只渲染首个），确定性、无泄漏；dev（`import.meta.env.DEV`，
  同 `r-icon` 约定）下 `console.error` 提示。注意该守卫在发布包里会被 DCE，故**面向消费者的
  兜底是「忽略行为」本身**（始终生效），告警只服务 ranui 自身开发/测试。

### docs：补齐 typed ref（消除 `closePopover` 强转）

- 迁移时对 `<r-popover>` 调 `closePopover()` 得手写 `as { closePopover?() }` 强转——其实
  组件类本就从 `ranui` 导出，`createRef<Popover>()` 即可拿到带方法的类型。属**可发现性缺口**，
  非缺失能力。BUILDER.md / CLAUDE.md 补了「typed ref 到自定义元素命令式方法」的写法。

### popover：修复 `ranui/content` 子路径解析失败

- **问题**：`package.json` 广告了 `./content` 导出（`-> ./dist/content.js`），但构建入口清单
  `componentEntries` 里没有 `content`，`dist/content.js` 从不产出——`import 'ranui/content'`
  在 rolldown/vite 下直接解析失败。`<r-content>` 实际由 `import 'ranui/popover'` 顺带注册，
  这条广告出来的入口是个纯陷阱。
- **修复**：把 `content` 加进 `componentEntries`（指向 `components/popover/content/index.ts`），
  使广告的入口真的产出、可 import。
