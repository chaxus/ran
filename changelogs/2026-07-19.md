# Changelog — 2026-07-19

## packages/ranui

来自 weread UI 迁移（去 React，改 builder + signal）的实战反馈，做两处让 builder 更贴近
AI 直觉、并修一个 export 陷阱。

### builder：`children()` 支持响应式 getter（reactive children）

- **动机**：此前 `children()` 只接静态节点，反应式的「条件子节点 / 列表」必须手写
  `createRef` + `createEffect` + `replaceChildren`，且首帧 `ref.current` 为 null 需早退——
  样板重复、易错，也不是 AI 写响应式 UI 的第一直觉（AI 会直接写
  `.children(() => cond() ? A : B)`）。
- **改动**：`ElementBuilder.children()` / `replaceChildren()` 与 `ShadowBuilder.children()`
  的每个参数现在都可以是 **getter `() => 节点 | 节点[] | null`**。getter 标记一个 live 区域：
  读到的 signal 变化时 getter 重跑，区域 DOM 就地重建（锚点 comment 定位，静态兄弟节点位置不变）。
  绑定的 effect 由当前 `createRoot` 拥有，随页面 dispose 自动清理。
- **SSR**：getter 只求值一次（静态快照），反应式是纯客户端行为。
- 静态用法完全不变（新签名是旧签名的超集）；新增 4 个浏览器用例 + 2 个 SSR 用例。
- 类型：新增并导出统一的 `Child` 类型（`ranui/builder` 与 `@/utils/builder` 均导出），
  取代内联联合。**注意命名**：是 `Child`（children 的参数类型），与 `r-input` / `InputBuilder` 无关。

### popover：修复 `ranui/content` 子路径解析失败

- **问题**：`package.json` 广告了 `./content` 导出（`-> ./dist/content.js`），但构建入口清单
  `componentEntries` 里没有 `content`，`dist/content.js` 从不产出——`import 'ranui/content'`
  在 rolldown/vite 下直接解析失败。`<r-content>` 实际由 `import 'ranui/popover'` 顺带注册，
  这条广告出来的入口是个纯陷阱。
- **修复**：把 `content` 加进 `componentEntries`（指向 `components/popover/content/index.ts`），
  使广告的入口真的产出、可 import。
