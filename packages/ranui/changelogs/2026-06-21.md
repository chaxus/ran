### 视觉回归测试体系建立

#### 背景与动机

项目已有 Argos CI + Playwright 的工具链，但测试粒度过粗：`visual/components.spec.ts` 截的是整个 demo 页面的 section，一个 tab 顺序变化就会让所有截图失效，且"select 文字没有垂直居中"这类具体布局 bug 根本抓不住。

本次目标：为每个组件建立**隔离挂载 + 精确断言**的视觉回归测试，同时补充布局关键属性的计算样式断言。

#### 新增 `test/e2e/helpers.ts` — 共享测试工具

- **`isolatedSetup(page, url, waitForTag)`**：导航到 dev server、等待指定 custom element 注册完毕、冻结所有 CSS 动画和 transition（`animation-duration: 0.001ms`）、清空 body。
- **`mount(page, html)`**：将组件 HTML 注入空白页面；custom element 定义已由 dev server 启动时注册，插入后同步升级。
- **`FREEZE_ANIMATIONS`** 常量：统一的动画冻结 CSS，所有 spec 共用。

设计原则：每个测试在空白页面上独立挂载目标组件，不依赖 demo 页面的布局和其他组件。

#### `test/e2e/select.spec.ts` — 特殊的布局精确断言

`select — with value` 测试包含一个**计算样式布局断言**，直接 document 已知 bug："select 组件文字没有垂直居中"：

```ts
// 用 Range API 拿到文字节点的实际渲染 bbox，
// 与 .selection 容器的 bbox 对比中心点差值
const centering = await page.evaluate(() => {
  const range = document.createRange();
  range.selectNodeContents(item);
  const textBox = range.getBoundingClientRect();
  const cBox = container.getBoundingClientRect();
  return { diff: Math.abs(textBox.top + textBox.height / 2 - (cBox.top + cBox.height / 2)) };
});
expect(centering!.diff).toBeLessThanOrEqual(3); // ≤ 3px 误差
```

该断言**当前会失败**（设计如此）：它文档化了已知 bug，修复 `.selection-item` 的垂直对齐后自动变绿，防止将来回归。截图断言放在布局断言之前，确保 `--update-snapshots` 时截图基准总能生成。

### 主题系统测试策略（分析记录）

#### CSS-only 主题的测试层次

讨论了针对 ranui 多主题（light/dark/system × 8 个 pack）的测试策略：

| 层次             | 工具             | 验证内容                                       | 现状            |
| ---------------- | ---------------- | ---------------------------------------------- | --------------- |
| 行为层           | Vitest           | setTheme/setThemePack API、localStorage 持久化 | ✅ 已有         |
| 静态契约         | Vitest（读文件） | 源码中是否声明了 token 名                      | ✅ 已有         |
| **运行时值**     | **Playwright**   | **CSS 变量在浏览器里实际 computed 的值**       | ❌ 缺失         |
| 视觉回归         | Playwright       | pack 的外观截图                                | ⚠️ 仅 button 行 |
| Dark × Pack 组合 | Playwright       | dark + wired / pixel-retro 等高风险组合        | ❌ 缺失         |

**核心结论**：静态分析（regex 检查源码）只能证明 token 被"写进"了文件，无法证明 CSS 变量在浏览器里 cascade 正确、computed 值符合预期。运行时值断言是目前缺失的关键层。

**组合爆炸问题**：3 themes × 8 packs × 20+ 组件 × 4 状态 = 1,920 组合，全量截图不可行。建议策略：选 3–5 个"参考组件"（button、input、select）覆盖所有 token 类别，在所有 pack 下截图；单独针对 dark × 高风险 pack 补充组合测试。

### Wired JS 主题的测试体系（核心工作）

#### 为什么 JS 主题特殊

wired pack 通过 roughjs 在 body 上叠加一个全页 SVG overlay，每个组件对应若干 `<path>`。与 CSS-only 主题的核心差异：

- **位置依赖真实布局**：`getBoundingClientRect()` 返回值在 jsdom 里永远为零，路径对齐问题只能在真实浏览器里测
- **异步生命周期**：`rAF + MutationObserver + ResizeObserver` 三个异步源，测试需精确控制 flush 时机
- **状态残留风险**：overlay SVG、WeakMap seeds、tracked Set 必须在 deactivate 时完整清理

#### 重写 `test/unit/wired-overlay.test.ts`（4 → 36 个测试）

原有 4 个测试仅覆盖基础路径。本次完整补齐 9 个 describe 块，共 36 个测试：

**`overlay SVG lifecycle`（6 个）**

- `activateWiredBorders` 向 body 追加带 `data-ran-wired-overlay` 标记的 SVG
- overlay 设置了 `aria-hidden="true"`（对屏幕阅读器透明）
- `deactivateWiredBorders` 将 SVG 元素**整体从 body 移除**（不只清空 path）
- 重复 deactivate 是安全 no-op
- 重新激活后 DOM 里只有一个 overlay（无重复）

**`path drawing`（5 个）**

- 有效 bbox 的元素生成 SVG paths
- 所有 path 携带 `fill="none"`、`stroke-linecap="round"`、`stroke-linejoin="round"`
- `width < 2` 的元素被跳过（无 paths）
- `height < 2` 的元素被跳过
- `width = 2 && height = 2` 不被跳过（边界值）
- path 的 `d` 属性包含 SVG 绘图命令（M/C/L）

**`element tracking`（4 个）**

- 多个元素各自拥有独立的 SVG `<g>` 组
- 重复调用 `activateWiredBorders` 不会双倍绘制
- 激活后**新增到 DOM 的元素**在下一次 MutationObserver flush 时自动获得边框
- 激活时 `ResizeObserver.observe` 被调用（用 class stub 替代 vi.fn 以支持 `new` 构造）

**`deactivation side-effects`（4 个）**

- deactivate 后 paths 全部清除
- deactivate 后 window resize 不触发重绘
- deactivate 后新插入元素不获得边框（MutationObserver 已断开）
- `ResizeObserver.disconnect()` 在 deactivate 时被调用

**`data-wired-skip`（2 个）**

- 设置 `data-wired-skip` 移除该元素的 paths
- skip 一个元素不影响同级元素的 paths（差值比较）

**`reactivation`（2 个）**

- deactivate → activate 后边框正确重绘
- 中间经历 pack 切换后 reactivate 仍然正确

**`component selector coverage`（3 个）**

- 12 个目标 tag（r-button, r-input, r-checkbox, r-select, r-modal, r-message, r-tab, r-progress, r-colorpicker, r-skeleton, r-card, r-section）全部被追踪
- 非目标 tag（r-icon, div, span）不被追踪
- 带 `data-wired-skip` 的目标 tag 被排除

**`pack attribute changes`（4 个）**

- 移除 pack 属性后 paths 清除
- `syncWiredBordersForThemePack` 在 pack=wired 时激活
- `syncWiredBordersForThemePack` 对所有非 wired pack（paper/pixel-retro/windows-98/neo-brutalism/default）均正确停用
- pack 属性缺失时也正确停用

**`installWiredThemePackSync`（5 个）**

- 后续 pack 切换触发激活
- 切换回非 wired 触发停用
- 重复调用 install 是幂等的（只有一个 overlay）
- `uninstallWiredThemePackSync` 后不再响应 pack 属性变化
- 指定自定义目标元素时只监听该元素，不响应 documentElement 上的变化

#### 关键修复：ResizeObserver stub 方式

原本写法 `vi.stubGlobal('ResizeObserver', vi.fn(() => ({ ... })))` 会报 `is not a constructor`——`vi.fn()` 包装箭头函数后不满足 `new` 调用要求。修复方式：

```ts
vi.stubGlobal(
  'ResizeObserver',
  class {
    observe = observeSpy;
    unobserve = vi.fn();
    disconnect = vi.fn();
    constructor(_cb: ResizeObserverCallback) {}
  },
);
```

同时在 `afterEach` 追加 `vi.unstubAllGlobals()` 防止 ResizeObserver 替换状态泄漏到后续测试（原 `vi.restoreAllMocks()` 不处理 stubGlobal）。

#### 新增 `test/e2e/wired-overlay.spec.ts` — 浏览器对齐测试

单元测试（jsdom）无法验证"path 是否真的压在组件上面"，因为 jsdom 的 `getBoundingClientRect()` 返回全零。补充 Playwright E2E 测试，在真实 Chromium 中验证：

- overlay SVG 存在于 document.body
- overlay 带 `aria-hidden="true"` 且 `pointer-events: none`
- paths 存在且携带正确 SVG 属性
- path 的包围盒覆盖对应组件的中心点（对齐断言）
- deactivate 后 overlay 从 DOM 完整移除
- 重复激活/停用后只有一个 overlay（无重复）
- 视觉截图：默认模式 + dark 模式下的 wired 外观

**对齐断言的实现思路**：

```ts
// 找到按钮的中心点
const componentCenterX = bBox.left + bBox.width / 2;
const componentCenterY = bBox.top + bBox.height / 2;

// 至少一个 SVG <g> 组的 bbox 应该覆盖该中心点
const covers = groups.some((g) => {
  const gBox = g.getBoundingClientRect();
  return (
    gBox.left <= componentCenterX &&
    componentCenterX <= gBox.right &&
    gBox.top <= componentCenterY &&
    componentCenterY <= gBox.bottom
  );
});
expect(covers).toBe(true);
```

## Demo 多语言支持

### 背景与动机

Demo 页面内容全部为英文，增加中英文切换功能，提升中文用户的体验和可读性。

### 实现方案

采用轻量级 `data-i18n*` 属性 + `applyLanguage()` 函数方案，不引入任何第三方 i18n 库。

#### 核心设计

**5 种属性类型对应 5 类内容：**

| 属性                      | 用途                                                          | 实现                              |
| ------------------------- | ------------------------------------------------------------- | --------------------------------- |
| `data-i18n="key"`         | 普通文本替换                                                  | `el.textContent = t[key]`         |
| `data-i18n-html="key"`    | 含 HTML 标签的文本（如 h1 中的 `<br>` `<em>`）                | `el.innerHTML = t[key]`           |
| `data-i18n-section="id"`  | r-section 自定义元素标题/副标题                               | `el.setAttribute('heading', ...)` |
| `data-i18n-card="prefix"` | r-card 自定义元素 title/description                           | `el.setAttribute('title', ...)`   |
| `data-i18n-group="key"`   | 组件分组标签（CSS `::before content: attr(data-group)` 驱动） | `el.dataset.group = t[key]`       |

**持久化：** 选择的语言存入 `localStorage`（key: `ran-lang`），刷新后自动恢复。

**r-section / r-card 的 Shadow DOM 透传：** 这两个组件通过 `attributeChangedCallback` 监听 `heading`/`subtitle`/`title`/`description` 属性变化并更新 Shadow DOM 内容。`applyLanguage()` 调用 `setAttribute()` 即可触发，无需特殊处理。

### 翻译词条

每种语言约 100 个翻译 key，覆盖：

- 导航栏链接
- Hero 区域（标题、副标题、CTA 按钮、统计数字、特性 pill）
- 工作台面板（工具栏、字段标签、trace 行）
- 主题包按钮标签
- 组件跳转链接
- 画廊分组标签
- Why / Themes / Components / Notes / Code / Style API 等各 section 的标题和副标题
- 所有组件 r-card 的 title、description、meta 标签
- 通用 "View notes / 查看文档" 链接
